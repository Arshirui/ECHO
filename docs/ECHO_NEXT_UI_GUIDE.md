# ECHO NEXT UI 指南

这份文档写给 ECHO NEXT 的界面开发、评审和后续维护人员。目标不是做一份漂亮口号，而是让每次改 UI 时都有明确判断标准：什么该放在界面上，什么该藏到诊断里，什么会拖慢播放，什么会让用户误解当前状态。

ECHO NEXT 的 UI 应该像一台安静、可靠、信息清楚的桌面音乐播放器。它可以精致，但不能花哨；可以有 HiFi 气质，但不能为了“发烧感”牺牲可读性、性能和稳定播放。

## 总方向

优先级从高到低：

1. 播放状态可信。
2. 大曲库下不卡。
3. 用户能看懂当前发生了什么。
4. 高风险操作有明确边界。
5. 视觉干净、克制、有质感。
6. 动效和装饰只服务理解，不抢注意力。

界面气质：

| 方向 | 应该这样做 | 不应该这样做 |
| --- | --- | --- |
| 桌面播放器 | 信息密度适中，列表、封面、播放栏稳定 | 做成营销页、社交 App 或纯视觉展板 |
| 本地曲库 | 强调文件、专辑、艺术家、队列和播放状态 | 把在线内容放到比本地播放更核心的位置 |
| HiFi | 清楚展示输出、采样率、DSP、ReplayGain、EQ 状态 | 用模糊词暗示 bit-perfect 或音质提升 |
| 长期维护 | 组件边界清楚，样式 token 化，可回归 | 临时堆全局 CSS、到处写硬编码颜色 |
| 性能稳定 | 虚拟列表、分页、懒加载、后台任务降载 | 大列表里直接重算、重渲染、跑重动画 |

## 用户心智

用户打开 ECHO NEXT 时最关心三件事：

1. 我现在能不能正常听歌。
2. 我的曲库、歌单、设置有没有被安全保存。
3. 如果出问题，我能不能知道是文件、输出设备、网络、歌词、MV 还是程序状态的问题。

所以 UI 不要把用户引向“猜”。每个页面都应该回答：

- 当前页面在管什么。
- 当前数据来自哪里。
- 当前操作会影响什么。
- 当前状态是否可恢复。
- 当前错误下一步应该看哪里。

## 应用框架

默认桌面壳由这些区域组成：

```text
ECHO 自定义标题栏
  左侧导航
  主内容区
  底部播放栏
```

### 标题栏

标题栏应该有 ECHO 自己的产品感，但仍然像桌面软件：

- 保留清楚的窗口控制区域。
- 快捷操作只放高频、低风险入口。
- 不在标题栏塞复杂设置。
- 窗口动作必须走 typed preload IPC，Renderer 不直接调用 Electron API。
- 发生全局错误时可以有轻量提示，但不要盖住播放控制。

### 左侧导航

导航承担“我现在在哪里”的责任：

- 主导航放曲库和播放相关入口，例如 `Songs`、`Albums`、`Artists`、`Queue`、`Playlists`。
- 扩展能力放在主导航后段，例如 `Streaming`、`Downloads`、`Cloud / Remote`、`Plugins`。
- 设置、收藏、历史等可以作为 utility 区域。
- 隐藏功能不能只隐藏按钮名，还要同步处理路由、搜索词、文案和入口。
- 不要为了展示功能数量把侧边栏变成工具箱。

### 主内容区

主内容区负责完成当前页面的核心任务：

- 列表页优先扫描效率。
- 详情页优先关系和上下文。
- 设置页优先稳定、分组和可回退。
- 诊断页优先证据和可复制信息。
- 空状态要给出下一步，不要只写“暂无数据”。

### 底部播放栏

播放栏是全局最高信任区域：

- 展示当前曲目、艺术家、封面、播放/暂停、进度、音量、队列入口。
- 进度条必须表达真实播放状态，不要用“看起来在动”的假状态掩盖问题。
- 如果音频实际在播但 UI 状态滞后，优先修 UI 状态合并和渲染层，不要先怀疑音频链路。
- 输出模式、设备、EQ、ReplayGain、变速等影响声音的状态要有可见入口或明确提示。
- 播放栏更新要避免带动全 App 重渲染。

## 页面规范

### Songs

`Songs` 是大曲库主战场，首要目标是快、稳、可扫描。

- 必须支持搜索、排序、选择、右键菜单和播放。
- 列表行高度稳定，不因封面、标签、错误提示而抖动。
- 大曲库必须分页或虚拟化。
- 封面使用缩略图，不直接加载大图。
- 搜索结果为空时说明当前筛选条件，而不是暗示曲库丢失。
- 批量操作要明确影响范围。

### Albums

`Albums` 负责专辑墙和专辑详情：

- 专辑墙优先封面、标题、专辑艺术家和年份。
- 详情页要清楚展示曲目顺序、disc、track number。
- 同名专辑或拆分专辑要给用户整理入口。
- 封面缺失时使用稳定占位，不要造成布局跳动。
- 大量封面懒加载，滚动时不要阻塞播放。

### Artists

`Artists` 负责艺术家聚合：

- 清楚区分 artist、albumArtist、参与艺术家等概念。
- 同名或重复艺术家不要用自动合并掩盖风险。
- 网络补全信息只能辅助展示，不应覆盖高可信本地标签。

### Queue

`Queue` 是临时播放顺序，不是长期歌单：

- 当前播放曲目要明显。
- 下一首、移除、拖拽排序等操作要即时反馈。
- 队列为空时引导用户从 `Songs`、`Albums` 或 `Playlists` 添加。
- 不要把队列页做成第二套曲库管理系统。

### Playlists / Liked / History

这些页面都围绕“用户长期行为”：

- `Playlists` 强调保存、命名、排序和批量管理。
- `Liked` 强调快速找回喜欢的歌。
- `History` 强调时间线和最近播放，不要默认暴露过多隐私信息。
- 删除或清空操作必须有确认和可理解后果。

### Lyrics

歌词页是沉浸页面，但仍然要可控：

- 当前行突出，前后文仍可读。
- 翻译、罗马音、假名增强、桌面歌词开关要边界清楚。
- 歌词时间偏移要易调、易回退。
- 背景图、模糊和动态效果不能压低文字可读性。
- 桌面歌词是独立界面，字体、锁定、穿透、翻译/罗马音状态不要随便和主歌词页混在一起。

### Downloads

下载页要让用户知道任务处于哪一步：

- 搜索、解析、下载、提取音频、导入曲库要有阶段状态。
- 失败时显示可操作原因，例如网络、工具缺失、平台限制、FFmpeg 失败。
- 下载不应该抢占播放链路资源；播放期间避免重任务并发。
- 涉及来源合法性和平台限制时文案要清楚，不要鼓励绕过限制。

### Cloud / Remote

远程源页面要保守：

- 新连接先测试，再保存。
- 大远程库默认小范围索引或仅浏览。
- 连接失败要区分认证失败、网络失败、路径失败和服务端不可用。
- 远程内容要有“来源”感，不要伪装成本地文件。

### Streaming

流媒体搜索是补充能力：

- 搜索结果要标来源、质量、时长和可用性。
- 试听和加入队列要明确是远程内容。
- 不要让流媒体入口压过本地曲库。
- 网络失败要可恢复，不要影响本地播放。

### Connect

投放和远程播放需要强调连接状态：

- 设备发现、连接中、已连接、失败、断开要清楚。
- 本机播放和远端播放状态不要混淆。
- 设备不可达时给出重试和刷新入口。

### Plugins

插件页必须安全优先：

- 插件权限要明确展示。
- 高权限能力，例如 `settings:write`、`library:write`、`network`，要明显提示风险。
- 插件错误不能拖垮主应用。
- 禁用、重载、查看日志要容易找到。

### Settings

设置页是高风险区域，要求比普通页面更稳：

- 分组清楚，不要把所有设置平铺。
- 保存过的旧配置必须 merge 默认值，不能因为新增字段白屏。
- 危险操作单独分区，并说明影响。
- 搜索结果不能泄露隐藏功能。
- 每个影响声音的设置都要说明是否影响 bit-perfect、DSP、音量或输出链路。
- 设置项的默认值要保守，优先稳定播放。

## 状态和反馈

常见状态要一致：

| 状态 | 界面应该表达 |
| --- | --- |
| Loading | 正在加载，尽量保留旧内容，不闪屏 |
| Empty | 当前确实没有数据，并给出下一步 |
| Scanning | 正在扫描曲库，显示范围和进度 |
| Playing | 音频链路正在播放，进度和按钮同步 |
| Paused | 用户或系统暂停，进度不假动 |
| Buffering | 等待数据或解码，不等同于失败 |
| Failed | 失败原因和下一步可见 |
| Degraded | 功能降级但主流程仍可用 |
| Disabled | 当前不可用，并说明前置条件 |

错误提示要避免三类问题：

- 只有“失败了”，没有原因。
- 只有内部错误码，用户不知道下一步。
- 把网络、文件、设备、权限、程序 bug 混成同一个提示。

## 文案规范

产品文案要短、准、诚实：

- 不夸大音质。
- 不承诺无法保证的网络结果。
- 不用“智能”“无损增强”这类模糊词掩盖真实行为。
- 不在普通界面解释架构细节。
- 不把开发者备忘录写进用户可见区域。
- 对危险操作直接说影响，例如“会清空本地曲库索引”，不要写成“优化数据库”。

推荐写法：

| 场景 | 推荐 | 避免 |
| --- | --- | --- |
| 空曲库 | 选择一个音乐文件夹开始扫描 | 这里什么都没有 |
| 输出失败 | 当前设备无法打开，已保留上一首播放状态 | 播放失败 |
| 网络候选 | 来自网络匹配，可能需要手动确认 | 已找到正确歌词 |
| DSP 状态 | EQ 已开启，输出不再是原始信号 | HiFi 增强中 |
| 危险操作 | 清空索引不会删除音乐文件，但会移除扫描记录 | 重置 |

## 视觉规范

### 颜色

新 UI 必须使用语义 token，而不是直接写死颜色。优先使用：

- `--theme-app-bg`
- `--theme-page-bg`
- `--theme-page-text`
- `--theme-heading-text`
- `--theme-muted-text`
- `--theme-panel-bg`
- `--theme-panel-bg-strong`
- `--theme-panel-border`
- `--theme-field-bg`
- `--theme-button-bg`
- `--theme-list-row-bg`
- `--theme-list-row-bg-hover`
- `--theme-list-row-bg-active`
- `--theme-player-bg`
- `--theme-control-bg`
- `--theme-accent-solid-bg`
- `--theme-danger-solid-bg`

详细规则见 [theme-contract.md](./theme-contract.md)。

不要新增硬编码白色、浅灰背景或只适配暗色主题的写法。确实是固定色资产时，必须用附近注释说明原因。

### 字体和密度

- 页面标题要克制，桌面工具不需要到处巨型标题。
- 列表、表格、设置项要优先可扫描。
- 按钮文字不要过长，长操作放到 tooltip、确认弹窗或说明区域。
- 中文、英文、日文、韩文混排时要检查行高和截断。
- 路径、错误、设备名这类长文本要可换行或可横向滚动，不能撑破布局。

### 图标

- 优先使用项目已有图标和 `lucide-react`。
- 图标按钮要有可访问名称或 tooltip。
- 危险操作图标和颜色要一致。
- 不要为普通操作手写临时 SVG，除非项目确实没有合适图标。

### 封面和图片

- 列表使用缩略图。
- 大封面只在详情页、歌词页、Now Playing 等必要场景加载。
- 缺封面时使用稳定占位。
- 网络图像失败时不要造成布局塌陷。
- 背景封面模糊只能作为辅助，不要影响歌词和按钮可读性。

### 动效

动效要轻：

- 允许按钮 hover、焦点、弹层淡入。
- 允许播放状态的轻量反馈。
- 大列表、大封面墙、歌词滚动不要使用昂贵动画。
- 不要让动画触发大量 layout/reflow。
- 用户快速切歌时，动效不能阻塞状态更新。

## 组件和样式边界

样式目前主要在：

- `src/renderer/styles/tokens.css`
- `src/renderer/styles/theme.css`
- `src/renderer/styles/layout.css`
- `src/renderer/styles/app.css`
- `src/renderer/styles/ui-polish.css`
- 页面或功能样式，例如 `songs.css`、`lyrics.css`、`queue.css`、`desktop-lyrics.css`

新增样式时：

- 通用 token 放 `tokens.css`。
- 主题映射放 `theme.css`。
- 应用框架布局放 `layout.css`。
- 页面专属样式优先放到对应功能 CSS。
- 不要把新页面的大量细节继续塞进一个全局文件。
- 不要依赖 `legacy-theme-bridge.css` 做新功能。

组件边界：

- 页面组件负责数据入口、页面状态和主要布局。
- 复用组件负责局部交互和展示。
- 播放 tick、扫描进度、下载进度等高频状态不要随便进入全局 React render path。
- Renderer 不直接做文件扫描、音频解码、封面提取等重任务。

## 性能红线

UI 不能为了外观牺牲播放稳定。

必须避免：

- 在 render 中对全曲库做过滤、排序、聚合。
- 播放进度每 tick 推动整个 App 重渲染。
- 滚动列表时加载原始封面大图。
- 大量 DOM 节点一次性渲染。
- 大列表上绑定昂贵 hover、阴影、滤镜或模糊。
- 播放期间启动不必要的全量远程索引、封面重建、网络补全。
- 页面切换时阻塞音频链路。

推荐做法：

- 列表分页或虚拟化。
- 派生数据提前缓存或在 store/service 层计算。
- 封面多尺寸缓存。
- 高频状态局部订阅。
- 搜索输入 debounce。
- 后台任务分批，播放期间降载。
- 失败保留旧 UI，不要白屏。

## 可访问性和键盘

桌面播放器也要照顾键盘和可读性：

- 主要按钮可 Tab 到达。
- 图标按钮有 `aria-label` 或等价描述。
- 输入框、滑块、开关有明确 label。
- 焦点样式可见。
- 危险确认弹窗默认焦点不要落在危险按钮上。
- 颜色不能是唯一状态表达，尤其是错误、警告、选中。

## 设置和兼容性

新增设置时必须考虑旧用户已有配置：

- 给默认值。
- 读取旧配置时 merge 默认结构。
- 缺字段不能导致设置页白屏。
- 删除或重命名设置要保留迁移或 fallback。
- 新增快捷键 action 时要合并默认 shortcut，而不是假设用户配置完整。

设置 UI 要清楚表达影响范围：

| 设置类型 | 必须说明 |
| --- | --- |
| 输出模式 | 影响设备、独占、稳定性或系统混音 |
| EQ / DSP | 会改变音频信号 |
| ReplayGain | 会改变响度，不是文件本身变化 |
| 歌词来源 | 自动匹配可能不准 |
| 下载目录 | 影响保存位置和导入结果 |
| 远程源 | 可能占网络、耗时、依赖服务端 |
| 插件权限 | 可能读取网络、写设置、改曲库 |

## 隐藏功能

隐藏功能要真正隐藏：

- 不出现在侧边栏。
- 不出现在设置搜索结果。
- 不出现在普通说明文案。
- 不出现在 tooltip、空状态和按钮标题里。
- 路由直达时要检查是否已解锁。
- 解锁状态要有兼容默认值，旧配置不能白屏。

不要只改按钮名，留下搜索 alias、设置分组或路由入口。

## 安全和高风险操作

高风险操作要分层：

| 操作 | UI 要求 |
| --- | --- |
| 删除曲目 | 说明是移出曲库还是删除真实文件 |
| 清空索引 | 说明不会删除音乐文件，但会移除扫描记录 |
| 重建数据库 | 提示备份和耗时 |
| 批量编辑标签 | 显示影响数量，允许取消 |
| 修改输出到独占 / ASIO | 说明可能无声、失败或占用设备 |
| 插件高权限 | 展示权限和来源 |
| 清理缓存 | 说明封面、临时文件或下载结果可能重新生成 |

任何会影响真实文件、数据库、播放稳定或用户长期数据的操作，都不能只有一个普通按钮。

## 国际化

当前界面中英文都要能承载：

- 新文案走 i18n，不要只写死英文。
- 中文不要机翻腔。
- 英文不要过长导致按钮撑开。
- 路由 label、description、tooltip、empty state、错误提示要同步。
- 开发文档可以中文优先，但用户可见界面要考虑已有语言体系。

## 评审清单

提交 UI 改动前至少过一遍：

1. 这个页面的主任务是否一眼能看懂。
2. 播放栏是否仍然稳定、可信。
3. 大曲库下是否会一次性渲染过多内容。
4. 是否新增了硬编码主题色。
5. Light / Dark / Follow System 是否都能看。
6. 空状态、加载、失败、禁用状态是否完整。
7. 旧设置缺字段时是否仍能打开页面。
8. 长标题、长路径、长设备名是否会撑破布局。
9. 网络失败、文件缺失、设备不可用是否有不同反馈。
10. 高风险操作是否有确认和影响说明。
11. 隐藏功能是否没有泄露入口和搜索词。
12. 新文案是否进入 i18n。

## 快速验证建议

文档或纯样式小改动通常只需要检查 diff 和页面截图。涉及功能交互时按改动范围选最小验证：

| 改动 | 建议验证 |
| --- | --- |
| 文案 / 文档 | 检查 diff |
| 单页面样式 | 打开对应页面，检查亮/暗主题和窗口宽度 |
| 播放栏 | 播放、暂停、切歌、进度、音量 |
| 曲库列表 | 大量数据、搜索、排序、选择、右键 |
| 设置页 | 旧配置兼容、切换、保存、重启后读取 |
| 歌词页 | 当前行、偏移、翻译/罗马音、桌面歌词 |
| 下载 / 远程 | 失败状态、取消、播放期间是否降载 |
| 主题 token | `npm run lint` 或至少运行主题颜色检查 |

不要因为改了一个小按钮就跑全量重测试。也不要因为只看起来是 UI，就忽略它对播放 tick、封面加载、后台任务和设置兼容性的影响。

## 一句话标准

如果一个 UI 改动让用户更清楚、更稳定、更少误判，同时没有增加播放压力，它就是好改动。反过来，如果它只是更炫、更满、更像功能展示墙，却让状态更难懂或更容易卡顿，就不适合 ECHO NEXT。
